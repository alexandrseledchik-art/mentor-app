# Interaction Principles v2

## Зачем нужен этот документ

Этот документ фиксирует новый принцип работы Telegram-first entry layer.

Цель не в том, чтобы сделать "умный роутер" или "универсальный чат".
Цель в том, чтобы бот вёл живой, короткий и честный разговор, который помогает:

- понять, что хочет пользователь;
- быстро отделить внешний скрининг от реальной бизнес-диагностики;
- задать один лучший следующий вопрос;
- привести пользователя к сильному результату:
  `запрос -> диагностика -> ограничение -> приоритет -> инструменты -> сохранённый результат`.

## Главный принцип

Не:

`classifier -> router -> template response`

А:

`listener -> diagnostic framing -> best next move`

Это означает:

- бот сначала слушает смысл запроса;
- потом собирает диагностическую рамку;
- потом выбирает один лучший следующий шаг;
- и только после этого ведёт в диагностику, инструмент или внешний скрининг.

## Роли слоёв

### 1. Entry layer

Entry layer не ставит диагноз и не симулирует глубокий консалтинг.

Он должен:

- принять свободный вход;
- понять, что именно пользователь хочет разобрать;
- зафиксировать цель, симптомы и пробел данных;
- выбрать лучший следующий шаг;
- сохранить непрерывность разговора.

Он не должен:

- ставить жёсткий диагноз при слабом сигнале;
- превращать каждый вход в полный diagnostic case;
- подменять живой диалог меню-вопросами;
- разбрасываться по нескольким сценариям сразу.

### 2. Router

Router — это диспетчер, а не консультант.

Router должен только выбрать следующий action:

- `capability`
- `website_screening`
- `tool_navigation`
- `ask_question`
- `diagnostic_result`

Router не должен:

- формулировать содержательные вопросы про бизнес;
- определять ограничение системы;
- сочинять "диагноз по одному сообщению";
- подменять диагностический prompt.

### 3. Diagnostic layer

Основная консультативная работа происходит внутри диагностического prompt-а.

Если данных ещё мало, система не притворяется, что уже готова к диагнозу,
а задаёт один лучший вопрос, который сильнее всего уменьшит неопределённость.

Planner должен:

- выявить цель клиента;
- выделить опорные симптомы;
- понять, чего не хватает для честной диагностики;
- задать один сильный уточняющий вопрос;
- либо пустить дальше в diagnostic core, если сигнала уже достаточно.

### 4. Diagnostic core

Diagnostic core включается только тогда, когда уже есть минимальная рамка:

- понятна цель хотя бы предварительно;
- есть 1–2 опорных симптома;
- есть достаточно сигнала для рабочих гипотез.

Diagnostic core работает по отдельному reasoning prompt и не должен запускаться по привычке или из-за наличия URL.

## Базовая логика разговора

### Шаг 1. Понять, что хочет пользователь

Первый вопрос системы:

не "в какой mode он попал?",

а:

- что пользователь хочет понять;
- что он хочет решить;
- какой результат он ожидает.

Если цель неясна, нужно уточнить именно её.

### Шаг 2. Отделить тип входа

Нужно различать:

#### A. Capability / meta

Пользователь спрашивает о возможностях бота, формате работы, голосовых, мини-аппе, хранении и т.д.

Правильное поведение:

- ответить прямо;
- не запускать диагностику;
- не строить гипотезы про бизнес;
- не создавать фальшивый case.

#### B. Website screening

Пользователь прислал только сайт или сайт + короткий контекст, но сигнала для диагностики бизнеса ещё нет.

Правильное поведение:

- сделать внешний скрининг;
- честно сказать, чего по сайту утверждать нельзя;
- не называть внутреннее ограничение бизнеса;
- завершить открытым вопросом о запросе пользователя.

#### C. Business diagnosis

Пользователь описывает реальную бизнес-ситуацию, цель, боль, симптомы.

Правильное поведение:

- пойти в диагностическую рамку;
- при нехватке сигнала задать один сильный вопрос;
- при достаточном сигнале вести в diagnostic core.

#### D. Tool navigation

Пользователь явно просит подобрать инструмент или объяснить, что использовать.

Правильное поведение:

- понять, есть ли уже достаточный контекст;
- если контекста мало, сначала уточнить задачу;
- не подменять диагностику каталогом инструментов.

## Как задавать вопросы

### Правильный вопрос

Хороший вопрос:

- уменьшает неопределённость;
- помогает развести 2–3 рабочие гипотезы;
- уточняет цель или опорный симптом;
- двигает к следующему диагностическому шагу.

### Неправильный вопрос

Плохой вопрос:

- шаблонный;
- не привязан к текущему кейсу;
- выглядит как развилка ради развилки;
- уводит в методологию, а не в запрос пользователя;
- не влияет на выбор ограничения или следующего шага.

### Правило одного вопроса

На каждом ходе задаётся один лучший вопрос, а не список из трёх развилок.

Если данных уже достаточно, лишний вопрос не задаётся.

## Как должна работать диагностическая рамка

Entry layer не обязан сразу строить весь диагноз, но должен двигаться в этой последовательности:

1. Цель клиента
2. Симптомы
3. Рабочие гипотезы
4. Главный пробел данных
5. Лучший следующий шаг

Это означает:

- сначала понять, чего человек хочет;
- потом зафиксировать, что именно болит;
- потом собрать 1–2 рабочие версии;
- потом спросить только то, что реально изменит следующую диагностическую развилку.

## Принцип свободы модели

Модель не должна быть зажата в набор предзаданных menu-style действий.

Но свобода не означает бесконтрольность.

Свобода модели должна жить внутри рамки:

- не врать про уверенность;
- не ставить диагноз без сигнала;
- не смешивать meta-вопрос и бизнес-диагностику;
- не генерировать "умный шум";
- не уходить от запроса пользователя.

Правильная свобода:

- живой язык;
- естественная формулировка вопроса;
- гибкость в следующем ходе;
- способность слышать реальный смысл запроса.

Неправильная свобода:

- произвольные догадки;
- псевдодиагноз при слабом сигнале;
- случайные развилки;
- ответ "по шаблону", который игнорирует контекст разговора.

## Telegram UX принципы

### 1. Не должно быть тишины

Если система думает, распознаёт голос или готовит ответ, пользователь должен видеть признак жизни:

- `typing`
- fallback при ошибке

### 2. Кнопки не должны затмевать следующий шаг

Если у ответа есть сохранённый разбор или скрининг, кнопка полезна.

Но кнопка не должна перетягивать смысл с главного действия пользователя.

После внешнего скрининга главный CTA — не кнопка, а продолжение разговора.

### 3. Следующий шаг должен быть очевиден

Пользователь всегда должен понимать:

- что система сейчас сделала;
- чего она не утверждает;
- что нужно сделать дальше.

## Сильные и слабые примеры

### Сильный пример

Пользователь:

`Хочу продать бизнес`

Правильный ответ:

- коротко зафиксировать цель;
- признать, что данных мало;
- задать один вопрос, который раскрывает стоп-фактор продажи:
  стоимость, упаковка, операционная зависимость, финансы, покупательский интерес.

### Слабый пример

Пользователь:

`Хочу продать бизнес`

Плохой ответ:

`Вам важнее понять корень проблемы или сразу подобрать инструмент?`

Почему это плохо:

- вопрос слишком общий;
- не связан с конкретной целью;
- не уменьшает неопределённость по кейсу;
- звучит как системная развилка, а не как мысль консультанта.

## Что не должно происходить

- Meta-вопрос не должен превращаться в business diagnosis.
- URL сам по себе не должен превращаться в внутренний диагноз бизнеса.
- Слабый сигнал не должен превращаться в "главное ограничение".
- Повторное сообщение не должно обнулять траекторию разговора.
- Router не должен подменять основной диагностический prompt.
- Кнопки не должны заменять живой следующий ход.

## Новый стандарт качества entry layer

Entry layer работает хорошо, если:

- пользователь чувствует, что его действительно слушают;
- система не уводит его в случайный шаблон;
- следующий вопрос выглядит уместным и сильным;
- слабый сигнал остаётся слабым сигналом;
- сильный сигнал быстро переводится в диагностику;
- внешний скрининг, meta-ответы и диагностика не смешиваются;
- разговор ощущается живым, но не хаотичным.

## Рабочий критерий для каждой новой правки

Перед добавлением нового правила или маршрута нужно спрашивать:

1. Это помогает лучше услышать запрос пользователя?
2. Это уменьшает неопределённость, а не увеличивает шаблонность?
3. Это двигает к диагностике, ограничению и приоритету?
4. Это сохраняет честность при слабых данных?
5. Это делает разговор живее, а не механистичнее?

Если ответ "нет", правило или ветку не нужно добавлять.
